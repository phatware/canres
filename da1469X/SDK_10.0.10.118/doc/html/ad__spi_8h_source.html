25 /*
26  *
27  * Terminology:
28  * Adapter: A middleware abstraction layer for using a controller. It provides the following services:
29  * - Arbitration in accessing the controller (between multiple masters, multiple tasks)
30  * - Configuration of the controller
31  * - Block sleep while the controller is in use
32  * Controller: Top level view of SPI peripheral, including the complete configuration needed for being fully functional (IOs, DMAs, Driver configuration)
33  * Driver: The Low Level Driver associated with the controller
34  * SPI: The controller name (e.g SPI1, SPI2)
35  *
36  * API example usage:
37  * ad_spi_init(): Called by the system on power up for initializing the adapter. It should NOT be called by the application
38  * ad_spi_io_config(): Called by the application for configuring the controller interface before powering up the external connected devices (e.g sensors)
39  *
40  * ad_spi_open(); Called by the application for starting a transaction. The sleep is blocked until ad_spi_close() is called. No other task or master can start a new transaction using the same controller instance
41  * ad_spi_write(); Blocking write - caller task will block until the resource becomes available
42  * ad_spi_read(); Blocking read - caller task will block until the resource becomes available
43  * ad_spi_reconfig();
44  * ad_spi_write_async(); Non blocking write - caller task should retry until function returns no error. It will then be notified when transaction is completed
45  * ad_spi_read_async(); Non blocking read - caller task should retry until function returns no error. It will then be notified when transaction is completed
46  * ad_spi_close(); Called by the application for releasing the resource. System is allowed to go to sleep (if no other module blocks sleep)
47  *
48  *
49  *
50  * */
